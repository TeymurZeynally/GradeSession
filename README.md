# GradeSession

GradeSession — веб-приложение для организации и проведения оценивания студентов на ГИА (Государственной Итоговой Аттестации).

## Цель проекта

Проект предназначен для цифровизации процесса оценивания студентов на ГИА: подготовки сессии оценивания, распределения ролей между секретарём и членами комиссии, сбора оценок по критериям, контроля прогресса и фиксации итоговых оценок после обсуждения комиссии.

## Что позволяет проект

GradeSession позволяет:

* создать сессию оценивания;
* добавить студентов, темы работ и комментарии;
* настроить критерии оценивания, диапазоны баллов и веса критериев;
* задать состав комиссии;
* выпустить ссылки-приглашения и QR-коды для участников;
* назначить участникам роли секретаря или члена комиссии;
* собирать оценки членов комиссии по каждому студенту;
* отслеживать прогресс заполнения оценок;
* закрыть сессию оценивания;
* после закрытия сессии зафиксировать финальные оценки по итогам обсуждения.

## Быстрый старт в Docker

Образ:

```bash
ghcr.io/teymurzeynally/gradesession:latest
```

### Вариант для быстрого запуска с хранением в памяти

Этот вариант подходит для демонстрации, локальной проверки и временных сессий.

```bash
docker run --rm \
  --name grade-session \
  -p 8080:8080 \
  -e ASPNETCORE_URLS=http://+:8080 \
  -e ASPNETCORE_ENVIRONMENT=Development \
  -e Storage__Provider=Memory \
  ghcr.io/teymurzeynally/gradesession:latest
```

После запуска приложение будет доступно по адресу:

```text
http://localhost:8080
```

> ⚠️ Важно: при `Storage__Provider=Memory` все данные хранятся только в памяти процесса приложения.
> После остановки контейнера, перезапуска приложения, пересоздания контейнера или деплоя на другой инстанс данные будут потеряны.
> Этот режим не подходит для production-сценариев и реальных экзаменационных сессий.

## Запуск с персистентным хранением через etcd

Для сохранения данных между перезапусками используйте `Storage__Provider=Etcd`.

Ниже пример `docker-compose.yml` с приложением и etcd. В этом варианте данные приложения сохраняются в Docker volume `grade-session-etcd-data`.

```yaml
services:
  etcd:
    image: gcr.io/etcd-development/etcd:v3.6.12
    container_name: grade-session-etcd
    restart: unless-stopped
    command:
      - /usr/local/bin/etcd
      - --name=grade-session-etcd
      - --data-dir=/etcd-data
      - --listen-client-urls=http://0.0.0.0:2379
      - --advertise-client-urls=http://etcd:2379
      - --listen-peer-urls=http://0.0.0.0:2380
      - --initial-advertise-peer-urls=http://etcd:2380
      - --initial-cluster=grade-session-etcd=http://etcd:2380
      - --initial-cluster-token=grade-session
      - --initial-cluster-state=new
      - --auto-compaction-mode=revision
      - --auto-compaction-retention=1000
      - --quota-backend-bytes=8589934592
    volumes:
      - grade-session-etcd-data:/etcd-data
    healthcheck:
      test: ["CMD", "etcdctl", "--endpoints=http://localhost:2379", "endpoint", "health"]
      interval: 5s
      timeout: 3s
      retries: 10
    networks:
      - grade-session

  gradesession:
    image: ghcr.io/teymurzeynally/gradesession:latest
    container_name: grade-session
    restart: unless-stopped
    depends_on:
      etcd:
        condition: service_healthy
    environment:
      ASPNETCORE_ENVIRONMENT: Production
      ASPNETCORE_URLS: http://+:8080

      Jwt__Issuer: grade-session
      Jwt__Audience: grade-session-client-app
      Jwt__SecretKey: dev-secret-key-change-me-grade-session-very-long
      Jwt__ExpirationDays: "90"

      Storage__Provider: Etcd
      Storage__Etcd__ConnectionString: http://etcd:2379
      Storage__Etcd__KeyPrefix: /grade-session/prod
      Storage__Etcd__UseInsecureChannel: "true"
    ports:
      - "8080:8080"
    networks:
      - grade-session

volumes:
  grade-session-etcd-data:

networks:
  grade-session:
    driver: bridge
```

Запуск:

```bash
docker compose up -d
```

> Для production рекомендуется использовать длинный случайный `Jwt__SecretKey`, закрыть etcd от внешнего доступа, настроить резервное копирование volume etcd и не использовать демонстрационные секреты.

## Переменные среды

В Docker и Docker Compose настройки ASP.NET Core задаются через двойное подчёркивание `__`.

Например:

```bash
Storage__Provider=Etcd
Jwt__SecretKey=...
RateLimiting__PerIp__PermitLimit=900
```

### Основные настройки приложения

| Переменная               | Значения / пример           | Как влияет                                                                                     |
| ------------------------ | --------------------------- | ---------------------------------------------------------------------------------------------- |
| `ASPNETCORE_ENVIRONMENT` | `Development`, `Production` | Определяет окружение. В `Development` доступен Swagger UI и применяются development-настройки. |
| `ASPNETCORE_URLS`        | `http://+:8080`             | Адрес и порт, на котором приложение слушает HTTP-запросы внутри контейнера.                    |

### Хранилище

| Переменная                          | Значения / пример     | Как влияет                                                                                                                          |
| ----------------------------------- | --------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `Storage__Provider`                 | `Memory` или `Etcd`   | Выбирает backend хранения данных.                                                                                                   |
| `Storage__Etcd__ConnectionString`   | `http://etcd:2379`    | Адрес etcd. Нужен только при `Storage__Provider=Etcd`.                                                                              |
| `Storage__Etcd__KeyPrefix`          | `/grade-session/prod` | Префикс ключей в etcd. Позволяет разделять окружения или несколько инсталляций в одном etcd.                                        |
| `Storage__Etcd__UseInsecureChannel` | `true` или `false`    | При `true` используется небезопасный канал без TLS. Для production с TLS используйте `false` и соответствующую инфраструктуру etcd. |
| `Storage__Etcd__Username`           | `gradesession`        | Имя пользователя etcd, если включена аутентификация.                                                                                |
| `Storage__Etcd__Password`           | `...`                 | Пароль etcd. Обязателен, если задан `Storage__Etcd__Username`.                                                                      |

### JWT-аутентификация

| Переменная            | Значения / пример          | Как влияет                                                                            |
| --------------------- | -------------------------- | ------------------------------------------------------------------------------------- |
| `Jwt__Issuer`         | `grade-session`            | Issuer, который записывается в JWT и проверяется при авторизации.                     |
| `Jwt__Audience`       | `grade-session-client-app` | Audience, который записывается в JWT и проверяется при авторизации.                   |
| `Jwt__SecretKey`      | длинная случайная строка   | Ключ подписи JWT. При изменении ключа ранее выданные токены станут недействительными. |
| `Jwt__ExpirationDays` | `90`                       | Срок действия токена в днях.                                                          |

### Приглашения

| Переменная                 | Значения / пример      | Как влияет                                                                                                                                                                          |
| -------------------------- | ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Invites__ClaimMode`       | `Reject` или `Replace` | Определяет поведение при попытке занять уже занятую роль. `Reject` отклоняет нового пользователя. `Replace` разрешает новому пользователю заменить прежнего участника в этом слоте. |
| `Invites__TokenByteLength` | `32`                   | Размер случайной token-части приглашения в байтах. Чем больше значение, тем выше энтропия токена и тем длиннее ссылка.                                                              |

### Ограничения запросов

| Переменная                                    | Значения / пример  | Как влияет                                                                                            |
| --------------------------------------------- | ------------------ | ----------------------------------------------------------------------------------------------------- |
| `RequestLimits__Enabled`                      | `true` или `false` | Включает или выключает ограничения Kestrel для входящих запросов.                                     |
| `RequestLimits__MaxRequestBodySizeBytes`      | `10485760`         | Максимальный размер тела запроса. Влияет, например, на импорт больших списков студентов или комиссии. |
| `RequestLimits__RequestHeadersTimeoutSeconds` | `15`               | Таймаут получения HTTP-заголовков.                                                                    |

### Rate limiting

| Переменная                                          | Значения / пример  | Как влияет                                                               |
| --------------------------------------------------- | ------------------ | ------------------------------------------------------------------------ |
| `RateLimiting__Enabled`                             | `true` или `false` | Включает или выключает rate limiting для API.                            |
| `RateLimiting__PerIp__PermitLimit`                  | `900`              | Количество API-запросов на IP за окно времени.                           |
| `RateLimiting__PerIp__WindowSeconds`                | `60`               | Размер окна rate limit для IP.                                           |
| `RateLimiting__PerIp__QueueLimit`                   | `30`               | Размер очереди запросов сверх лимита для IP.                             |
| `RateLimiting__PerUser__PermitLimit`                | `600`              | Количество API-запросов на авторизованного пользователя за окно времени. |
| `RateLimiting__PerUser__WindowSeconds`              | `60`               | Размер окна rate limit для пользователя.                                 |
| `RateLimiting__PerUser__QueueLimit`                 | `30`               | Размер очереди запросов сверх лимита для пользователя.                   |
| `RateLimiting__AnonymousAuth__PermitLimitPerMinute` | `5`                | Лимит создания анонимных пользователей в минуту с одного IP.             |
| `RateLimiting__AnonymousAuth__PermitLimitPerHour`   | `30`               | Лимит создания анонимных пользователей в час с одного IP.                |
| `RateLimiting__AnonymousAuth__QueueLimit`           | `0`                | Размер очереди для anonymous auth-запросов сверх лимита.                 |

## Ролевая модель

В системе есть несколько уровней доступа.

### Создатель сессии

Создатель сессии — пользователь, который создал сессию оценивания.

Может:

* создать сессию;
* просматривать созданную сессию;
* выпускать ссылки-приглашения;
* перевыпускать ссылки-приглашения;
* видеть список приглашений по сессии.

Важно: создатель сессии и секретарь — не одно и то же. Для выполнения функций секретаря пользователь должен занять слот с ролью `Secretary`.

### Секретарь (`Secretary`)

Секретарь — участник сессии, который перешёл по ссылке-приглашению для слота секретаря.

Может:

* просматривать сессию;
* видеть прогресс заполнения оценок по членам комиссии;
* закрыть сессию оценивания;
* после закрытия сессии просматривать сводку оценок;
* сохранять финальные оценки по итогам обсуждения комиссии.

Секретарь не выставляет оценки за членов комиссии.

### Член комиссии (`CommitteeMember`)

Член комиссии — участник сессии, который перешёл по ссылке-приглашению для своего слота комиссии.

Может:

* просматривать назначенную сессию;
* видеть студентов и критерии;
* выставлять оценки по критериям;
* выставлять свою итоговую оценку студенту;
* отметить, что студент не оценивается;
* просматривать итоговые данные, если у него есть доступ к сессии.

Член комиссии может изменять оценки только пока сессия активна. После закрытия сессии оценки членов комиссии блокируются.

## Основные бизнес-процессы

### 1. Создание пользователя

При первом входе клиент создаёт анонимного пользователя и получает JWT access token.
Дальше запросы к API выполняются с этим токеном.

### 2. Создание сессии оценивания

Создатель задаёт:

* название сессии;
* диапазон финальной оценки;
* список студентов;
* темы работ и комментарии;
* критерии оценивания;
* минимальные и максимальные баллы по критериям;
* веса критериев;
* состав комиссии и роли участников.

После создания сессия сразу становится активной.

### 3. Выпуск приглашений

Создатель сессии выпускает приглашения для всех слотов комиссии.

Для каждого слота создаётся отдельная ссылка с токеном. Токен хранится на сервере только в виде хеша, поэтому открытую ссылку нужно скопировать или экспортировать сразу после выпуска.

При перевыпуске приглашений старые активные ссылки отзываются.

### 4. Принятие приглашения

Участник открывает ссылку-приглашение и занимает соответствующий слот:

* `Secretary` — секретарь;
* `CommitteeMember` — член комиссии.

Если слот уже занят, поведение зависит от `Invites__ClaimMode`:

* `Reject` — новый пользователь не сможет занять уже занятый слот;
* `Replace` — новый пользователь заменит прежнего участника в этом слоте.

### 5. Оценивание студентa членами комиссии

Член комиссии заполняет по каждому студенту:

* присутствие / статус оценивания;
* оценки по критериям;
* свою итоговую оценку.

Оценки валидируются по диапазонам критериев и диапазону итоговой оценки.
Если студент отмечен как `NotPresent`, оценки по критериям и итоговая оценка для этого студента не сохраняются.

### 6. Контроль прогресса

Секретарь видит прогресс по каждому члену комиссии:

* сколько студентов уже начато;
* сколько студентов полностью заполнено;
* сколько оценок по критериям заполнено;
* сколько итоговых оценок заполнено.

Это помогает понять, готова ли комиссия к закрытию сессии.

### 7. Закрытие сессии

Секретарь закрывает сессию после завершения оценивания.

После закрытия:

* оценки членов комиссии больше нельзя изменять;
* становится доступна сводка оценок;
* можно перейти к фиксации финальных оценок после обсуждения.

### 8. Финальные оценки после обсуждения

После закрытия сессии секретарь заполняет финальные оценки студентов по итогам обсуждения комиссии.

Финальная оценка проверяется по диапазону, заданному в настройках сессии.

## Статусы сессии

| Статус   | Значение                                                                                          |
| -------- | ------------------------------------------------------------------------------------------------- |
| `Draft`  | Черновик. Зарезервирован в модели, но текущий процесс создания сразу переводит сессию в `Active`. |
| `Active` | Сессия открыта, члены комиссии могут выставлять оценки.                                           |
| `Closed` | Сессия закрыта, оценки членов комиссии заблокированы, доступны сводка и финальные результаты.     |

## Безопасность и эксплуатационные замечания

* Не используйте демонстрационный `Jwt__SecretKey` в production.
* Храните `Jwt__SecretKey` и пароли etcd в секретах CI/CD или Docker secrets, а не в репозитории.
* Не публикуйте порт etcd наружу.
* Делайте резервные копии volume etcd.
* Для production-хранения используйте `Storage__Provider=Etcd`, а не `Memory`.
* При смене `Storage__Etcd__KeyPrefix` приложение начнёт читать и писать данные в другое пространство ключей.
* При смене `Jwt__SecretKey` все существующие пользовательские токены перестанут проходить проверку.

## Локальная разработка

В режиме `Development` приложение использует development-настройки, включая хранение в памяти и dev JWT-настройки. Swagger UI доступен только в development-окружении.

Примеры локальных URL:

```text
http://localhost:5269
https://localhost:7149
```

Для Docker-контейнера обычно используется:

```text
http://localhost:8080
```
